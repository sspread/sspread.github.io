SQL and the Seek to Know NoSQL

August 7.2014

SQL and NoSQL are different types of databases. SQL has been around since the 1970's and remains the dominant database structure. The term NoSQL ("Not Only SQL") was coined in 1998 and seeks to compliment SQL where it falls short. The big difference between the two is that SQL is relational and NoSQL is not. While there is much more to the story, I'm going to run with this idea for the benefit of high level comparison.

SQL

A relational database is one that stores information in a two-dimensional table format. Think about an Excel spreadsheet. The columns represent data fields, or attributes. The rows represent records that fill in the information for each field. For example a 'users' table, or relation, might hold fields for 'name', 'email' and 'dob'. Each row would be an individual user, uniquely identified by a primary key 'id' field. But that's not all. There could be another table with another set of attributes that relate to the original 'users' table. Maybe a schema example will illustrate more clearly.

This theoretical Twitter database schema represents a one-to-many relationship between two tables. Each unique user can write many tweets. Each unique tweet relates to one unique user. Many-to-many and one-to-one relationships also exist between tables, and all three relationships can be present in one schema. As you can imagine, this web of relations can get complicated in big data applications. SQL's scalability drawback has coaxed NoSQL into existence. But before we discuss that, let's look at a key feature of SQL.

In addition to offering an intuitive framework for storing data, SQL offers excellent data integrity, which is defined as having ACID:

  • Atomicity - All fields are updated at once, or none are updated at all.
  • Consistency - All transactions transform the state according to the integrity constraints in place.
  • Isolation - Multiple operations in a single transaction (i.e. updating multiple fields for a single record) do not intermingle in time with other transaction operations.
  • Durability - Completed transactions are safely and reliably stored on a hard drive.

NoSQL

The fact is, social media generates bigillafucks of data every day. As opposed to scaling up a SQL database by adding memory and faster hardware, NoSQL scales out by spreading the load over many commodity systems. It's design is relatively simple and cost effective. While SQL relational databases are birds of a feather, there are four types of NoSQL databases:

  • Key-values Stores - Unique keys point to data items. Simple, but inefficient for updating parts of a value.
  • Column Family Stores - Keys that point to multiple columns. Good for data distributed over multiple machines.
  • Document Databases - Documents are collections of other key-value collections. Makes for more efficient queries.
  • Graph Databases - Graph structures with nodes (i.e. entities) properties (information about nodes) and edges (connections between nodes). Efficient processing and capable for many machines.

It looks like the first 3 items are all key-value type databases, and the fourth is something else with which to be reckoned. As you may have guessed, the key-value databases map more easily to object-oriented programming, which is another +1 for NoSQL. On the downside, NoSQL generally lacks the ACID of relational databases. Whats more, the NoSQL data structure is more application-dependent, rendering it vendor incompatible and potentially unextendible to other applications.

Less technical thoughts...

Researching and writing this post was a little overwhelming because I have very little experience with databases. The pros and cons and nuances borne in the different types of databases runs deep. My goal was to provide a rudimentary, but correct and easy-to-understand comparison of SQL and NoSQL. Relational databases make more sense to me, maybe because I'm used to working in spreadsheets. And, the whole reason why SQL lacks scalability is admittedly a murky concept. But I look forward to working with both of them in class in the coming weeks.
