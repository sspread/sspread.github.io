Attacking Regular Expressions

Regular expressions can frighten new programmers because they resemble monsters with teeth made from backslashes and forward slashes. In this journey, I aim to befriend this beast by learning about it.

A regular expression (regex or regexp) defines a pattern by which to compare a string. In a world cluttered with too much information (TMI), it isn't difficult to appreciate the parsing power of regexes. As I researched regexes, they became less scary, and, I had a realization: there are a ton of resources on the web that can explain regexes better and more thoroughly than yours truly. So, in an effort to make the most of your time, I've made this blog post a potpourri of miniature Aha! moments, potential pitfalls and other big ideas that I discovered in my quest for understanding basic regex.

  • Ruby's /.../and %r{...}are the same thing. They are delimiters that surround the regex to tell Ruby that what's inside is, in fact, a regex. Other programming languages may have different syntax. Simply knowing /.../makes reading regex syntax more manageable, but this notation necessitates escaping any other forward slashes in the regex sequence.
  • The escape character \is perhaps the foremost toothy culprit for regex's unsightliness. Most characters in regex are literal, meaning that they seek to match themselves in the string. But, some characters are special metacharacters in regex; they represent a regex rule. The \simply turns a special character into a literal character. The next bullet point provides an example of how the \can also escape a literal character into a special character!
  • Metacharacters (i.e \w, \sand \d) act as shorthand equivalents for the POSIX bracket expressions, similar to character classes mentioned below, denoted with double brackets and colons (i.e. [[:word:]], [[:space:]]and [[:digit:]], respectively). These bracket expressions also match non-ASCII characters.
  • A character class, established with brackets [ ], is like a whole basket of characters that regex can use to make a match. Regex finds a match if any of the characters in the character class match the string's character.
  • Ruby uses =~and #matchto make match magic. =~returns the character index where the match began, while #matchreturns a MatchData object, which is a package of results of a pattern match, including the original regexp! Both methods return nilfor no match, and neither method cares what comes first: regex or string.
  • The caret ^is a special character (anchor) that dictates that the match must come at the beginning of the string. Otherwise, the match could begin anywhere. Similarly, the dolla sign $is an anchor for the end of the string. But take note! The ^inside the [ ]of a character class acts instead as a NOT operator, matching any character except those in the character class.
  • Regex is greedy when it comes to repetition. Quantifiers (i.e. *or +) allow regex to match the character more than zero or more than one time, respectively. By default, even once the regex matches the string, regex will keep matching characters along the string until it reaches a character that would kill the match. Placing a ?after these quantifiers will make it lazy, that is, limit the repetition to zero or one. Let's conclude with an example... The dot .means "any character".

    /.*te.*/.match("This regex is definitely greedy.")
    => #<MatchData "This regex is definitely greedy.">
    /.*te.*?/.match("This regex is definitely greedy.")
    => #<MatchData "This regex is definite">

The regex concepts beyond what I've discussed here I'm going to deem "intermediate". I hope my rambling is helpful as a supplement to fill in the gaps when learning regex. The Ruby Docs are a big-time resource for regex in the context of Ruby.

SQL and the Seek to Know NoSQL

SQL and NoSQL are different types of databases. SQL has been around since the 1970's and remains the dominant database structure. The term NoSQL ("Not Only SQL") was coined in 1998 and seeks to compliment SQL where it falls short. The big difference between the two is that SQL is relational and NoSQL is not. While there is much more to the story, I'm going to run with this idea for the benefit of high level comparison.

SQL

A relational database is one that stores information in a two-dimensional table format. Think about an Excel spreadsheet. The columns represent data fields, or attributes. The rows represent records that fill in the information for each field. For example a 'users' table, or relation, might hold fields for 'name', 'email' and 'dob'. Each row would be an individual user, uniquely identified by a primary key 'id' field. But that's not all. There could be another table with another set of attributes that relate to the original 'users' table. Maybe a schema example will illustrate more clearly.

This theoretical Twitter database schema represents a one-to-many relationship between two tables. Each unique user can write many tweets. Each unique tweet relates to one unique user. Many-to-many and one-to-one relationships also exist between tables, and all three relationships can be present in one schema. As you can imagine, this web of relations can get complicated in big data applications. SQL's scalability drawback has coaxed NoSQL into existence. But before we discuss that, let's look at a key feature of SQL.

In addition to offering an intuitive framework for storing data, SQL offers excellent data integrity, which is defined as having ACID:

  • Atomicity - All fields are updated at once, or none are updated at all.
  • Consistency - All transactions transform the state according to the integrity constraints in place.
  • Isolation - Multiple operations in a single transaction (i.e. updating multiple fields for a single record) do not intermingle in time with other transaction operations.
  • Durability - Completed transactions are safely and reliably stored on a hard drive.

NoSQL

The fact is, social media generates bigillafucks of data every day. As opposed to scaling up a SQL database by adding memory and faster hardware, NoSQL scales out by spreading the load over many commodity systems. It's design is relatively simple and cost effective. While SQL relational databases are birds of a feather, there are four types of NoSQL databases:

  • Key-values Stores - Unique keys point to data items. Simple, but inefficient for updating parts of a value.
  • Column Family Stores - Keys that point to multiple columns. Good for data distributed over multiple machines.
  • Document Databases - Documents are collections of other key-value collections. Makes for more efficient queries.
  • Graph Databases - Graph structures with nodes (i.e. entities) properties (information about nodes) and edges (connections between nodes). Efficient processing and capable for many machines.

It looks like the first 3 items are all key-value type databases, and the fourth is something else with which to be reckoned. As you may have guessed, the key-value databases map more easily to object-oriented programming, which is another +1 for NoSQL. On the downside, NoSQL generally lacks the ACID of relational databases. Whats more, the NoSQL data structure is more application-dependent, rendering it vendor incompatible and potentially unextendible to other applications.

Less technical thoughts...

Researching and writing this post was a little overwhelming because I have very little experience with databases. The pros and cons and nuances borne in the different types of databases runs deep. My goal was to provide a rudimentary, but correct and easy-to-understand comparison of SQL and NoSQL. Relational databases make more sense to me, maybe because I'm used to working in spreadsheets. And, the whole reason why SQL lacks scalability is admittedly a murky concept. But I look forward to working with both of them in class in the coming weeks.

Proc Party

Blocks, Procs and lambdas are instrumental function syntax in Ruby. After doing some due dilligence, I find it proper to extend to you my basic understanding of these guys. The differences between these closures are rather subtle and I sometimes wish there was only one type. But there are at least three. All three belong to the Procclass, as in procedure. This is very important.

Blocks

Blocks are syntax literals for Proc objects. As such, you can create a block with curlies {}(single line) or with do...end. The code example below illustrates two ways to call the block from a method, and feed it an argument.


block_call {|n| puts "I'm in the #{n}."}
=> I'm before the block.
=> I'm in the block.
=> I'm after the block.

Blocks are easy to code on the fly, but there's another way to create Procs that allows us to save our snippets to objects.

Procs

Although blocks actually are Procs, Procs behave a little differently when explicitly created with Proc.new.


proc_call(where_r_u)
puts
proc_call_vip(where_r_u)
=> I'm before the proc.
=> I'm in the proc, dummy!
=> I'm in the proc, fool!
=> I'm in the proc, doofus!
=> I'm after the proc.
=>
=> I'm before the proc.
=> I'm in the VIP section, dummy!
=> I'm in the VIP section, fool!
=> I'm in the VIP section, doofus!
=> I'm after the proc.

Our where_r_uProc is a little rude, but he's DRY because we only have to define him once and can use him forever.

Lambdas

Finally, creating a Proc with the keyword lambdamakes for even a slightly different Proc. Lambdas have two key differences from the Procs I described above. The first is that a lambda actually cares about the number of arguments passed to it. ¡Mira!

proq = proc {|x,y| "#{x} y #{y}"}
lamb = lambda {|x,y| "#{x} y #{y}"}

puts proq.call('uno','dos')
=> uno y dos
puts proq.call('uno')
=> uno y
puts lamb.call('uno','dos')
=> uno y dos
puts lamb.call('uno')
=> 'block in (main)': wrong number of arguments (1 for 2) (ArgumentError)

On second thought, using Spanish with these particular variable letters may have added unfair befuddlement to this example, but I hope you get it. lambdawill throw a fuss if you neglect an argument, and Proc.new(equivalent to proc) will throw your shit into a bag and push it down the stairs. Because it don't care.

The other difference is in the way lambdas and Procs treat return. lambdareturns only from itself, while procreturns from itself and from the method from whence it came. Looksee!


puts who_r_u ima_lambda
puts
puts who_r_u ima_lambda, 1
=> Who are you Procs?
=> I'm a lamb, duh!
=> Who's your friend?
=> I'm a Proc!
=>
=> Who are you Procs?
=> I'm a Proc!

When the lambda is executed first within #who_r_u, the rest of the method's code is executed. However, when the Proc is executed first, its returnskips the remainder of the method's code. Also worth noting in this example, Ruby does not allow arguments containing return, so I had to define ima_procwithin our method. However, lambdas can magically get away with it, so ima_lambdais defined outside the method and passed in as an argument.

Dealing a Poker Hand in Ruby

In this piece, I'm going to run through a short Ruby program I wrote to deal a hand of poker. I will discuss classes and instance variables, among other concepts. Let's take it step by step.

First, we must define a new class PokerHandas a blueprint to contain our methods and characterisics pertaining to a hand of poker. The first method in a new class is always going to be #initialize. When you create a new instance of the class PokerHand (i.e. an individual hand), you call #newon the class, which automatically looks for and executes #initializewithin the class definition. Let's do that now.

hand1=PokerHand.new(5)
=> There are 5 players.

We put a 5as the argument when creating the new class, which is interpreted by #initializeas the number of players in our game. Behind the scenes, #initializealso created a deck of cards, shuffled them, and dealt two to each player! I know, you want to see what you were dealt, right? Let's write another method to access the array value of the :herohash key.


hand1.hero_hand
=> Your hand is ["6c", "6s"]

Not bad, pocket 6's! See the '@' attached to our @playershash variable? That makes it an instance variable that our program can access across methods in our new PokerHand class, so long as it pertains to the same instance (i.e. hand1). Let's skip the betting for now and see a flop!


hand1.flop
=> Flop: ["Ac", "3s", "Jc"]

Probably not the flop we wanted to see, but our program is working! Let's write one more method to see our opponent's hands.


hand1.villain_hand(:two)
=> private method 'villain_hand' called for #(PokerHand:0x2c92248) (NoMethodError)

Not so fast! I told Ruby that #villain_handis private. This means that only our program can access this method and see player two's hand (or any of our opponent's hands, for that matter). We might have our program call this private method later on when we extend it to find out who wins, but that's for another day.

Ruby's The Enumerables

An enumerable is collection of information that is able to be enumerated. To enumerate means to process the elements in the collection, piece by piece. You might feel like an enumerable at the DMV.

Now, Ruby has its own idea of enumerables, not unlike our English definition above. But, it's a little confusing when you first learn it, as I myself am doing along with you. Let me ramble off my understanding now. This is not for you; it is for me. Ahem...

If you ask ruby what class Enumerable is Enumerable.class, it will return Module. It is used as a mixin in the Array class. Modules group together methods, classes and constants. Modules are similar to classes in this way, but you cannot create objects based on modules. Modules provide a namespace and allow you to share the functionality of instance methods between classes (mixins). Enumerator is a Class class. The Array class returns an Enumerator object when no block is given to #each. The Enumerator allows us to chain methods together indefinitely, as the previous chained methods will not be given blocks.

OK back to what we really came here to do. Let's discuss the Enumerable#group_by method. #group_by groups a collection by the criteria provided in the block. It returns a hash where each key represents a group whose values are an array containing its members.

An example...

fish_names = ["Nemo", "Bruce", "Marlin", "Glut", "Dory", "Coral", "Blenny"]
fish_names.group_by{|name| name.length}
=> {4=>["Nemo", "Glut", "Dory"], 5=>["Bruce", "Coral"], 6=>["Marlin", "Blenny"]}

#group_by grouped the names by their length, as specified in the block. Now let's omit the block and see what ruby returns...

fish_names.group_by
=> #(Enumerator: ["Nemo", "Bruce", "Marlin", "Glut", "Dory", "Coral", "Blenny"]:group_by)

Now we could store that Enumerator as a variable and save it for later. What use is this #group_by Enumerator without any grouping criteria? I have no idea.

JavaScript in Web Development

JavaScript is a computer language commonly used in web development. To understand JavaScript's role in the web, we must first establish a concept called client-side programming. The client is the computer that communicates with the server. You might think of the client as your laptop. Instead of only pointing to a file, the web server can run a program process the file before sending it to the client. This makes the web more dynamic. Alternatively, the client can run that program to process the file after it receives it. This client-side program is written in JavaScript. Let's look at some features and pitfalls.

We like JavaScript!

  • JavaSript is fast. It executes on the client side and reduces strain on the web server and saves bandwith.
  • JavaScript is compatible. It interacts with the user's web browser, so it is supported on any platform that supports that browser.
  • JavaScript is easy. It uses the DOM model so it can already recognize the objects on web pages.
  • JavaScript is extendible. Developers can write snippets that users can download that will interact with certain web pages and enhance functionality.

Not so fast...

  • JavaSript can be dangerous. Developers can write JavaScript programs that execute on the client's computer to do something that the user did not intend. The program could attack the client's system and/or send out private information. Although browsers limit JavaScripts, capabilities, malicious developers periodically find ways to circumvent the rules.
  • JavaScript can be inconsistent across browsers. Because JavaScript relies on the end user, developers cannot perfectly tailor their programs to varying browsers, settings and general environments.

Learning CSS - Padding, Border and Margin

The skeleton of a web site is constructed from HTML. Instead of bones, HTML defines all of its elements as boxes. Then comes CSS to flesh out the web site and make it beautiful. CSS uses the box model to wrap layers around the box defined by HTML. Each of these layers has unique properties.

Content is at the center of the box model. We can see content in blue. Content is the box that holds holds the site's text and images.

Padding creates a space around the content. Padding takes the background color defined by the content. The brown highlights that appear around the menu items on this page upon hover are created by padding.

Border surrounds the padding and content. It takes the color and design (i.e. solid, dotted, rounded corners) defined by the content. The blue underline on the header of this page is created with a border surrounding only the bottom of the header box.

Margin establishes an invisible force field around the whole package. The blank space above the navigation bar and on either side of the main content of this page are created by margins.

Note: the differences between padding, border and margin can be be unclear because they can be used interchangeably at times. For example, I could indent the title of this page "Tech" by using left margin or left padding. This is possible because there is no border around the content and we are not dealing with different background colors.

Finally, the thickness of each layer begins at the inner layer and extends outwards. When designing a web site, remember that the actual height and width of an element is the sum of the content, plus the layers added by padding, border and margin.
